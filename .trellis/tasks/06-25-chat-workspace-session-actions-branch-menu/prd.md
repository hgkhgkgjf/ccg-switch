# Chat 工作目录切换、会话菜单与 Git 分支操作

## Goal

让 Chat 新建会话也能明确选择项目工作目录，而不是只能依赖历史会话所在项目；同时补齐项目、会话、Git 分支的右键/下拉操作，使聊天工作区、历史会话和 Git 状态在同一界面里可见、可切换、可管理。

## Background

- 现有 `ChatSessionSidebar` 可以从 `get_dashboard_projects` 加载历史项目，并通过 `selectedProjectPath` 给 `onNewSession` 传入工作目录。
- 现有 `ChatPage` / `useChatStore` 的 `currentCwd` 是发送、`@` 文件补全、工具打开文件、工作区状态检测的核心上下文。
- 现有 `ContextBar` 只显示 Git 分支芯片，不能切换项目目录或操作分支。
- 新建会话在“没有任何历史会话 / 未选中项目 / 空白初始态”下容易缺少明确的工作目录入口。
- 后端已有 `chat_workspace_status(cwd)` 读取 Git root 和当前分支，但没有列分支、创建并检出分支、打开资源管理器、会话操作等命令。

## Requirements

### R1 工作目录切换器

- 在用户输入框上方展示当前工作目录，位置对齐参考图中的底部状态条：文件夹图标 + 项目名/路径。
- 点击工作目录可打开项目选择菜单，展示最近/已知项目，并支持“Open folder...”选择任意本地文件夹。
- 切换工作目录必须更新 `currentCwd`，并影响后续新会话发送、`@` 文件补全、`chat_workspace_status`、工具打开文件的相对路径解析。
- 工作目录菜单需要支持空状态：刚启动、没有历史项目、没有历史会话、`currentCwd = null` 时仍可显示稳定入口，并允许用户选择文件夹。
- 工作目录显示应优先用项目名/目录名，tooltip 展示完整路径。

### R2 项目右键菜单

- 项目列表项右键打开上下文菜单。
- 支持：置顶项目、在资源管理器中打开、创建永久工作树、重命名项目、全部标为已读、归档对话、移除。
- 第一阶段允许将缺少后端持久语义的操作做成禁用或无副作用占位，但 UI 不能误导用户认为已完成不可逆数据变更。
- “在资源管理器中打开”必须调用后端安全命令打开现有目录，路径不存在时给出错误提示。

### R3 会话右键菜单

- 会话列表和最近会话行右键打开上下文菜单。
- 支持：置顶会话、重命名会话、归档会话、标记为未读、在资源管理器打开、派生到本地、派生到新工作树。
- 重命名应复用/对齐已有 ai-bridge `session-titles-service.cjs` 的自定义标题能力，避免只改前端内存。
- 置顶/未读/归档若当前没有后端持久契约，应在设计中明确存储位置与扫描过滤规则后再实现。
- 资源管理器打开应优先定位会话源文件 `sourcePath`；如果路径不可用，回退到项目目录。

### R4 Git 分支菜单

- 在输入框上方同一上下文栏展示 Git 分支，点击可打开分支菜单。
- 分支菜单展示当前仓库分支列表，当前分支有选中标记。
- 支持创建并检出新分支；分支名需要校验，禁止空值、控制字符、明显危险参数和会被 Git 拒绝的名称。
- 非 Git 工作目录时显示非仓库状态或隐藏分支菜单，不能报错。
- 分支切换/创建后刷新 `chat_workspace_status`，让显示与后续发送上下文一致。

### R5 i18n、可访问性与主题

- 新增所有可见文案同步 `src/locales/en.json` 和 `src/locales/zh.json`。
- 图标按钮需要 `title` / `aria-label`。
- 菜单使用项目既有 DaisyUI + lucide-react 风格，支持亮/暗主题。
- 文本必须在窄宽度下截断或换行，不得挤压输入框和发送控件。

## Acceptance Criteria

- [ ] 空白初始态没有历史会话时，输入框上方仍显示工作目录入口；用户可以打开菜单并选择文件夹。
- [ ] 切换工作目录后，新建会话和第一次发送使用新目录，`@` 文件补全从新目录读取文件。
- [ ] 工作目录菜单展示已知项目、当前项目选中态和 “Open folder...” 入口。
- [ ] 项目右键菜单包含约定操作；已实现操作可用，尚未实现的持久化操作不会伪装成成功。
- [ ] 会话右键菜单包含约定操作；重命名至少能持久影响后续扫描出的会话标题。
- [ ] “在资源管理器中打开”对项目目录和会话文件可用，路径不存在时有错误反馈。
- [ ] Git 分支菜单在 Git 仓库中显示分支列表，当前分支有选中标记。
- [ ] 创建并检出新分支成功后，分支芯片更新为新分支；非法分支名不会执行 Git。
- [ ] 非 Git 目录、无目录、空项目列表、空会话列表均不崩溃。
- [ ] 相关单测先 RED 后 GREEN：`useChatStore` 空态/切目录、`chatSessionSidebarUtils` 菜单/排序、`chatWorkspaceStatus` 或新 Git 工具、上下文栏渲染。
- [ ] `npm test -- <related tests>`、`npm run build`、`cargo test --manifest-path src-tauri/Cargo.toml <related modules>`、`cargo check --manifest-path src-tauri/Cargo.toml` 通过。

## Out of Scope

- 不实现真实 Git merge/rebase/delete branch。
- 不删除真实历史会话文件，除非后续明确确认删除语义。
- 不改变 ai-bridge 会话流式协议。
- 不新增远程 Git 操作或网络访问。

## Open Questions

- 置顶、未读、归档、项目重命名等元数据应存储在 `~/.codemoss`、应用数据库，还是 provider 原生历史目录旁的 sidecar 文件？建议第一阶段优先使用应用自有元数据存储，避免污染 Claude/Codex 原生历史。
- “派生到本地 / 派生到新工作树”是否要求真实复制/创建工作树并打开新会话，还是先只生成目标目录/命令草稿？建议拆到第二阶段或先做禁用入口，避免本轮一次性引入高风险文件系统操作。
